## Overview

After publishing the configuration file with `php artisan vendor:publish --tag=tapix-config`, you will find it at `config/tapix.php`. This page documents every available option grouped by purpose.

## File Upload

### `max_file_size`

Maximum upload file size in bytes. Files exceeding this limit are rejected before processing begins.

- **Default:** `52428800` (50 MB)

```php [config/tapix.php]
'max_file_size' => 50 * 1024 * 1024,
```

Adjust this value based on your server's `upload_max_filesize` and `post_max_size` PHP settings. The lower of the three values applies.

### `max_rows`

Maximum number of rows allowed in a single import file. The wizard rejects files that exceed this count during the upload step.

- **Default:** `100000`

```php [config/tapix.php]
'max_rows' => 100_000,
```

For very large imports, consider increasing this alongside appropriate queue timeout values.

## Queue Processing

### `chunk_size`

Number of rows processed per queued job. Each chunk is dispatched as a separate job, allowing parallel processing and reducing memory usage.

- **Default:** `500`

```php [config/tapix.php]
'chunk_size' => 500,
```

Lower values reduce memory consumption per job but increase the total number of jobs. Higher values process faster but require more memory.

### `queues`

Queue names used for import processing jobs. Tapix uses separate queues for validation and execution, allowing you to prioritize or isolate each stage.

- **Default:** `['validation' => 'sync', 'execution' => 'default']`

```php [config/tapix.php]
'queues' => [
    'validation' => env('TAPIX_QUEUE_VALIDATION', 'sync'),
    'execution' => env('TAPIX_QUEUE_EXECUTION', 'default'),
],
```

By default, validation runs synchronously (`sync`) during the review step for immediate feedback. Execution runs on the `default` queue. To use dedicated queues:

```env [.env]
TAPIX_QUEUE_VALIDATION=imports-validation
TAPIX_QUEUE_EXECUTION=imports
```

```bash [Terminal]
php artisan queue:work --queue=imports-validation,imports
```

### `job_timeout`

Maximum execution time in seconds for each chunk processing job. If a job exceeds this duration, the queue worker terminates it.

- **Default:** `600` (10 minutes)

```php [config/tapix.php]
'job_timeout' => 600,
```

Your queue connection's `retry_after` value must exceed `job_timeout` to prevent overlapping job execution. For example, if `job_timeout` is 600, set `retry_after` to at least 630.

::note
The `retry_after` value is configured on your queue connection in `config/queue.php`, not in the Tapix configuration. Ensure it exceeds the `job_timeout` for whichever connection Tapix uses.
::

### `job_tries`

Maximum number of attempts for each chunk processing job before it is marked as failed.

- **Default:** `3`

```php [config/tapix.php]
'job_tries' => 3,
```

### `job_backoff`

Delay in seconds between retry attempts. Accepts an integer for a fixed delay or an array for progressive backoff.

- **Default:** `[10, 30]`

```php [config/tapix.php]
'job_backoff' => [10, 30],
```

With the default value, the first retry waits 10 seconds and the second retry waits 30 seconds.

## Import Store

### `store.path`

Directory path where Tapix stores per-import SQLite files. Each import gets its own SQLite database for row storage, which allows fast row-level operations without impacting your main database.

- **Default:** `storage_path('app/tapix')`

```php [config/tapix.php]
'store' => [
    'path' => storage_path('app/tapix'),
],
```

### `store.remote_disk`

When running queue workers on multiple servers, set this to a shared filesystem disk (e.g., S3) so workers can access import data across machines. The SQLite file is uploaded to this disk after the review step and downloaded by the worker before execution.

- **Default:** `null` (local filesystem only)
- **Environment variable:** `TAPIX_STORE_DISK`

```php [config/tapix.php]
'store' => [
    'remote_disk' => env('TAPIX_STORE_DISK'),
],
```

## Database

### `table_prefix`

Prefix applied to all database tables created by Tapix. This prevents naming conflicts with your application's existing tables.

- **Default:** `'tapix_'`

```php [config/tapix.php]
'table_prefix' => 'tapix_',
```

If you change this value after running migrations, you must manually rename the existing tables or run a fresh migration.

::warning
Changing the table prefix after running migrations will cause Tapix to look for tables that do not exist. You must manually rename the existing tables in your database or run a fresh migration to apply the new prefix.
::

## Multi-Tenancy

The `tenant` configuration group controls multi-tenant scoping. When enabled, all imports are automatically scoped to the current tenant.

### `tenant.enabled`

Enable or disable tenant scoping.

- **Default:** `false`

```php [config/tapix.php]
'tenant' => [
    'enabled' => false,
],
```

### `tenant.column`

The database column name used to store the tenant identifier on import records.

- **Default:** `'tenant_id'`

```php [config/tapix.php]
'tenant' => [
    'column' => 'tenant_id',
],
```

### `tenant.model`

The fully qualified class name of your tenant model. Set this to the Eloquent model that represents a tenant in your application.

- **Default:** `null`

```php [config/tapix.php]
'tenant' => [
    'model' => null,
],
```

When using Filament's built-in multi-tenancy, Tapix resolves the current tenant from the Filament panel automatically. Set the model here only if you need to override that behavior.

## Class Swapping

Tapix lets you swap the default models and job classes with your own implementations. This is useful for adding custom behavior to the import pipeline without modifying the package internals.

### Models

```php [config/tapix.php]
'models' => [
    'import' => \Tapix\Core\Models\Import::class,
    'failed_row' => \Tapix\Core\Models\FailedImportRow::class,
],
```

Replace these with your own model classes that extend the originals:

```php [config/tapix.php]
'models' => [
    'import' => \App\Models\CustomImport::class,
    'failed_row' => \App\Models\CustomFailedImportRow::class,
],
```

Tapix resolves model classes through `Tapix::importModel()` and `Tapix::failedRowModel()`, ensuring all internal queries use your custom models.

### Jobs

```php [config/tapix.php]
'jobs' => [
    'execute' => \Tapix\Core\Jobs\ExecuteImportJob::class,
    'validate' => \Tapix\Core\Jobs\ValidateColumnJob::class,
    'resolve_matches' => \Tapix\Core\Jobs\ResolveMatchesJob::class,
],
```

Replace these to customize how rows are executed, validated, or matched:

```php [config/tapix.php]
'jobs' => [
    'execute' => \App\Jobs\CustomExecuteImportJob::class,
    'validate' => \Tapix\Core\Jobs\ValidateColumnJob::class,
    'resolve_matches' => \Tapix\Core\Jobs\ResolveMatchesJob::class,
],
```

Job classes are resolved via `Tapix::executeJobClass()`, `Tapix::validateJobClass()`, and `Tapix::resolveMatchesJobClass()`.

## File Readers

### `readers`

A map of file extensions to reader classes. Tapix uses these to determine how to parse an uploaded file based on its extension.

- **Default:**

```php [config/tapix.php]
'readers' => [
    'csv' => \Tapix\Core\Importing\Readers\CsvReader::class,
    'txt' => \Tapix\Core\Importing\Readers\CsvReader::class,
    'xlsx' => \Tapix\Core\Importing\Readers\XlsxReader::class,
],
```

You can register custom reader classes for additional file formats. Each reader must implement the `Tapix\Core\Contracts\FileReader` interface. See the [Custom Readers](/features/file-formats) section for details.

## Custom Fields

### `custom_fields_adapter`

An optional adapter class that integrates Tapix with a custom fields or dynamic attributes system. When set, the import wizard displays custom fields alongside standard model attributes.

- **Default:** `null`

```php [config/tapix.php]
'custom_fields_adapter' => null,
```

The adapter must implement the `Tapix\Core\Contracts\CustomFieldsAdapter` interface.

## Notifications

### `notifications.enabled`

Whether to send a notification when an import completes (or fails).

- **Default:** `true`

```php [config/tapix.php]
'notifications' => [
    'enabled' => true,
],
```

### `notifications.class`

The notification class dispatched when an import finishes. You can replace this with a custom notification to change the message content, channels, or recipients.

- **Default:** `Tapix\Core\Notifications\ImportCompletedNotification::class`

```php [config/tapix.php]
'notifications' => [
    'class' => \Tapix\Core\Notifications\ImportCompletedNotification::class,
],
```

The custom notification class receives the completed `Import` model as a constructor argument.

## Cleanup

### `cleanup_after_days`

Number of days to retain completed import records and their associated temporary data. The `tapix:cleanup` scheduled command deletes records older than this value.

- **Default:** `30`

```php [config/tapix.php]
'cleanup_after_days' => 30,
```

To enable automatic cleanup, schedule the command in your application:

```php [routes/console.php]
use Illuminate\Support\Facades\Schedule;

Schedule::command('tapix:cleanup')->daily();
```

## Importer Discovery

### `importer_directories`

An array of directory paths where Tapix looks for importer classes. The `discoverImporters()` method on `TapixPlugin` scans these directories to register importers automatically.

- **Default:** `[app_path('Importers')]`

```php [config/tapix.php]
'importer_directories' => [
    app_path('Importers'),
],
```

Add additional directories if you organize importers across multiple locations, such as within domain-specific folders.

### `importer_namespace`

The PHP namespace corresponding to the default importer directory. This is used when generating new importer classes with the `make:tapix-importer` Artisan command.

- **Default:** `'App\\Importers'`

```php [config/tapix.php]
'importer_namespace' => 'App\\Importers',
```

## Routes

Tapix can register its own routes for handling file uploads and import progress outside of the Filament panel.

### `routes.enabled`

Whether Tapix should register its own routes.

- **Default:** `true`

```php [config/tapix.php]
'routes' => [
    'enabled' => true,
],
```

Set to `false` if you only use Tapix through the Filament panel and do not need standalone routes.

### `routes.prefix`

The URL prefix for Tapix routes.

- **Default:** `'imports'`

```php [config/tapix.php]
'routes' => [
    'prefix' => 'imports',
],
```

### `routes.domain`

An optional domain constraint for Tapix routes. When set, routes are only accessible on this domain.

- **Default:** `null`

```php [config/tapix.php]
'routes' => [
    'domain' => null,
],
```

### `routes.middleware`

Middleware applied to all Tapix routes.

- **Default:** `['web', 'auth']`

```php [config/tapix.php]
'routes' => [
    'middleware' => ['web', 'auth'],
],
```

Add additional middleware as needed, such as `verified` or a custom authorization middleware.

## Publishable Assets

Tapix makes several asset groups available for publishing:

| Tag | Description |
|-----|-------------|
| `tapix-config` | Configuration file (`config/tapix.php`) |
| `tapix-migrations` | Database migration files |
| `tapix-views` | Blade view templates (for UI customization) |

```bash [Terminal]
# Publish config (included in standard installation)
php artisan vendor:publish --tag=tapix-config

# Publish migrations (only if you need to customize them)
php artisan vendor:publish --tag=tapix-migrations

# Publish views (only if you need to customize the UI)
php artisan vendor:publish --tag=tapix-views
```
