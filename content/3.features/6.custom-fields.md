Tapix supports importing data into custom (user-defined) fields through a pluggable adapter system. This is useful for applications like CRMs where end users can create their own fields beyond the standard model attributes.

## The CustomFieldsAdapter Contract

To integrate custom fields, implement the `CustomFieldsAdapter` interface:

```php
interface CustomFieldsAdapter
{
    public function getFields(string $modelClass): ImportFieldCollection;

    public function getRelationshipFields(string $modelClass): ImportFieldCollection;

    public function saveValues(Model $record, array $values, int|string|null $tenantId = null): void;

    public function clearValues(Model $record, array $fieldCodes, int|string|null $tenantId = null): void;
}
```

### Methods

- **`getFields()`** -- Returns an `ImportFieldCollection` of additional field definitions for the given model class. These fields are merged with the importer's standard fields during the mapping step.

- **`getRelationshipFields()`** -- Returns relationship fields defined in the custom fields system. These are automatically converted to entity links, enabling relationship linking during import.

- **`saveValues()`** -- Persists custom field values to the database after the primary record has been saved. Receives the saved model instance, an associative array of custom field values, and the optional tenant ID.

- **`clearValues()`** -- Removes custom field values when the CSV contains explicit blank cells during update imports. Receives the model instance, an array of field codes to clear, and the optional tenant ID.

## Registration

Register your adapter using the `Tapix` facade in a service provider:

```php [app/Providers/AppServiceProvider.php]
use Tapix\Core\Tapix;

Tapix::useCustomFieldsAdapter(\App\Imports\MyCustomFieldsAdapter::class);
```

When no adapter is registered, custom field functionality is disabled and only the importer's standard fields are used.

## Relaticle Custom Fields Integration

Tapix ships a first-party adapter for the [Relaticle Custom Fields](https://relaticle.github.io/custom-fields/) package. When both packages are installed, the adapter is **auto-discovered** -- no configuration needed.

```bash [Terminal]
composer require relaticle/custom-fields
```

That's it. Tapix detects `relaticle/custom-fields` at boot and registers the `RelaticleCustomFieldsAdapter` automatically. Custom fields defined in Relaticle appear in the import wizard alongside standard fields.

### How Auto-Discovery Works

During application boot, Tapix checks if the `Relaticle\CustomFields\CustomFieldsPlugin` class exists. If it does and no adapter has been explicitly registered, the Relaticle adapter is used. To override auto-discovery with your own adapter, call `Tapix::useCustomFieldsAdapter()` in your service provider -- explicit registration always takes priority.

### Field Type Mapping

The adapter maps Relaticle field types to Tapix import field types:

| Relaticle type | Tapix FieldType |
|---|---|
| text, url, phone, email | Text / Email |
| textarea, rich-editor | Text |
| number, percentage | Number |
| currency | Currency |
| date | Date |
| datetime | DateTime |
| boolean, toggle | Boolean |
| select, radio | Choice |
| multi-select, tags | MultiChoice |
| record (lookup) | Relationship field |

Unmappable types (color picker, file upload) fall back to Text since CSV import is inherently text-based.

## Per-Importer Adapter Override

For applications with multiple custom field backends, individual importers can override the global adapter:

```php
class ContactImporter extends BaseImporter
{
    public function customFieldsAdapter(): ?string
    {
        return \App\Imports\CrmCustomFieldsAdapter::class;
    }
}
```

The resolution order is:

1. `$importer->customFieldsAdapter()` (per-importer override)
2. `Tapix::useCustomFieldsAdapter()` (explicit global registration)
3. Auto-discovered adapter (Relaticle detected)
4. `null` (no custom fields)

## How It Works

### Field Merging

The `BaseImporter::allFields()` method merges standard importer fields with any fields returned by the adapter's `getFields()` method. This means custom fields appear alongside standard fields in the column mapping step without any changes to your importer class.

### Internal Key Prefixing

Custom field keys are prefixed with `custom_fields_` internally to distinguish them from standard model attributes. This prevents naming collisions between custom fields and built-in columns.

### Batch Processing

During `ExecuteImportJob`, custom field data is collected for each chunk of rows and flushed to the adapter at the end of each chunk. Both `saveValues()` and `clearValues()` are called once per record, reducing the number of database operations compared to saving values row by row.

### Clearing Values

When a CSV contains blank cells for custom fields during an update import, those field codes are passed to `clearValues()`. This allows users to explicitly remove custom field values by leaving cells empty in the CSV.

### Relationship Fields

Relationship fields returned by `getRelationshipFields()` are automatically converted to entity links. This allows custom relationship fields to participate in the same relationship linking workflow as standard entity link fields, including match resolution and record creation.

## Use Case

A typical use case is a CRM application where administrators define custom fields per entity type (contacts, companies, deals). The adapter reads the field definitions from a `custom_fields` table and maps them to `ImportField` instances. During import execution, standard attributes are saved to the model directly, while custom field values are routed through the adapter's `saveValues()` method to a separate storage table.
