Tapix includes built-in support for multi-tenant applications. When enabled, all imports are automatically scoped to the current tenant, and tenant context is preserved across queue jobs.

## Configuration

Enable multi-tenancy in the Tapix config file:

```php [config/tapix.php]
'tenant' => [
    'enabled' => false,
    'column' => 'tenant_id',
    'strict' => true,
    'model' => null,
],
```

| Option | Default | Description |
|--------|---------|-------------|
| `enabled` | `false` | Whether tenant scoping is active. |
| `column` | `tenant_id` | The foreign key column on the imports table. |
| `strict` | `true` | When enabled, queries return zero results if no tenant context is set. See [Strict Mode](#strict-mode). |
| `model` | `null` | The tenant model class (e.g., `App\Models\Team`). Used by adapters that need to resolve tenant models. |

## Custom Tenant Resolver

Register a custom tenant resolver in a service provider:

```php [app/Providers/AppServiceProvider.php]
use Tapix\Core\Tapix;

Tapix::resolveTenantUsing(function () {
    return auth()->user()?->current_team_id;
});
```

The resolver should return the tenant ID (integer or string) for the current request context.

## Shipped Middleware

Tapix ships a `SetTenantContext` middleware for the most common tenant patterns. Add it to your routes or middleware groups:

```php
use Tapix\Core\Http\Middleware\SetTenantContext;

// User ID as tenant (default)
Route::middleware(SetTenantContext::class)->group(function () {
    // ...
});

// Custom attribute (e.g., user's team_id)
Route::middleware(SetTenantContext::class . ':team_id')->group(function () {
    // ...
});
```

When no attribute is specified, the middleware uses the authenticated user's primary key as the tenant ID. When an attribute is specified (e.g., `team_id`), it reads that attribute from the user model instead.

For more complex tenant resolution logic, use `Tapix::resolveTenantUsing()` instead.

## TenantContextService

The `TenantContextService` manages tenant state throughout the request lifecycle and across queue jobs.

### Setting the Tenant

```php
use Tapix\Core\Services\TenantContextService;

// Set a custom resolver closure
TenantContextService::setTenantResolver(function () {
    return auth()->user()?->current_team_id;
});

// Set the tenant ID directly
TenantContextService::setTenantId(42);
```

### Reading the Tenant

```php
// Resolve the current tenant ID
// Resolution order: custom resolver -> Context facade -> null
$tenantId = TenantContextService::getCurrentTenantId();

// Check if a tenant context exists
if (TenantContextService::hasTenantContext()) {
    // ...
}
```

### Temporary Context Switching

Use `withTenant()` to execute a closure within a different tenant context:

```php
TenantContextService::withTenant($otherTenantId, function () {
    // All queries and operations here are scoped to $otherTenantId
});
// Original tenant context is restored after the closure
```

### Resetting State

For testing or cleanup, use `reset()` to clear all tenant state:

```php
// Clear both the custom resolver and stored tenant ID
TenantContextService::reset();

// Or clear just the resolver
TenantContextService::clearResolver();
```

## Strict Mode

When strict mode is enabled (default), queries on tenant-scoped models return **zero results** if no tenant context has been set. This prevents accidental data leakage across tenants.

```php [config/tapix.php]
'tenant' => [
    'enabled' => true,
    'strict' => true, // default
],
```

Without strict mode, forgetting to set tenant context would silently return all records regardless of tenant. Strict mode makes this a safe default -- you get empty results instead of everyone's data.

### Bypassing Strict Mode

For admin panels, CLI commands, or seeder scripts that intentionally need access to all records, bypass the scope on specific queries:

```php
use Tapix\Core\Models\Import;
use Tapix\Core\Models\Scopes\TenantScope;

// Returns all imports regardless of tenant
$allImports = Import::withoutGlobalScope(TenantScope::class)->get();
```

Or disable strict mode globally via config when you need unscoped queries as the default:

```php [config/tapix.php]
'tenant' => [
    'strict' => false,
],
```

## Automatic Query Scoping

When tenancy is enabled, the `TenantScope` global scope is applied to the `Import` model. This ensures that all Eloquent queries on imports are automatically filtered by the current tenant:

```php
// Only returns imports belonging to the current tenant
$imports = Import::all();
```

No manual `where` clauses are needed. The scope is applied transparently.

## Filament Panel Integration

When your Filament panel has tenancy configured, `TapixPlugin` automatically registers the `SetTenantContextMiddleware`. This middleware sets the tenant context at the beginning of each request, so imports created through the panel are always associated with the correct tenant.

::note
When using `TapixPlugin` in a Filament panel with tenancy, tenant context middleware is registered automatically. No additional setup is required beyond enabling tenancy in the Tapix config and configuring your panel's tenancy as usual.
::

## Queue Jobs

Import queue jobs use the `TenantAware` trait to serialize and restore tenant context. When a job is dispatched, the current tenant ID is captured. When the job executes on a worker, the tenant context is restored before any database operations run.

When strict mode is enabled and a job is dispatched without tenant context, a `TenantContextMissingException` is thrown at execution time. This catches configuration mistakes early rather than silently processing records without tenant scoping.

## Laravel Octane

Tapix automatically cleans up the static tenant resolver between Octane requests. The `TenantContextService` listens for `RequestTerminated` events and calls `clearResolver()` to prevent resolver state from leaking between requests. The tenant ID itself is stored in Laravel's Context facade, which Octane already resets between requests.
